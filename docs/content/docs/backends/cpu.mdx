# CPU Backend

The CPU backend provides optimized execution on x86_64 and ARM64 processors. It's always available and serves as a fallback when GPU backends are unavailable.

## Features

- **SIMD Vectorization**: AVX2, AVX-512 (x86), NEON (ARM)
- **Multithreading**: Automatic parallelization with thread pools
- **Cache Optimization**: Memory access patterns optimized for L1/L2/L3 cache
- **BLAS Integration**: Optional Accelerate (macOS), OpenBLAS, or MKL

## Using CPU Backend

```cpp
#include <mlx/mlx.h>
using namespace mlx::core;

// Explicitly use CPU
set_default_device(Device::cpu);

// Or specify per-operation
auto a = random::normal({1000, 1000}, float32, Device::cpu);
auto b = matmul(a, a, Device::cpu);
```

## Thread Configuration

### Setting Thread Count

```cpp
// Get current thread count
int threads = cpu::get_num_threads();
std::cout << "Using " << threads << " threads\n";

// Set thread count
cpu::set_num_threads(8);

// Use all available cores
cpu::set_num_threads(std::thread::hardware_concurrency());
```

### Thread Affinity

```cpp
// Bind threads to specific cores (Linux)
cpu::set_thread_affinity({0, 1, 2, 3});  // Use cores 0-3

// Use performance cores only (if hybrid architecture)
cpu::prefer_performance_cores(true);
```

## SIMD Optimization

### Automatic Vectorization

Operations automatically use best available SIMD instructions:

```cpp
// Vectorized element-wise operations
auto a = random::normal({1000000});
auto b = random::normal({1000000});
auto c = add(a, b);  // Uses AVX2/AVX-512/NEON
```

### Check SIMD Capabilities

```cpp
auto caps = cpu::get_capabilities();
std::cout << "AVX2: " << caps.avx2 << "\n";
std::cout << "AVX-512: " << caps.avx512 << "\n";
std::cout << "NEON: " << caps.neon << "\n";
std::cout << "SVE: " << caps.sve << "\n";
```

## BLAS Integration

### Accelerate Framework (macOS)

```cpp
// Automatically uses Accelerate on macOS
auto a = random::normal({1000, 1000});
auto b = matmul(a, a);  // Uses vecLib/BLAS
```

### OpenBLAS

Build with OpenBLAS for optimized linear algebra:

```bash
cmake -B build \
    -DMLX_BUILD_ACCELERATE=OFF \
    -DMLX_BUILD_OPENBLAS=ON \
    ..
```

### Intel MKL

Build with Intel MKL for best x86 performance:

```bash
cmake -B build \
    -DMLX_BUILD_ACCELERATE=OFF \
    -DMLX_BUILD_MKL=ON \
    ..
```

## Memory Optimization

### Cache-Friendly Patterns

```cpp
// Good: Sequential memory access (cache-friendly)
auto a = random::normal({1000, 1000});
auto row_sums = sum(a, 1);  // Sum along rows

// Less optimal: Strided access
auto col_sums = sum(a, 0);  // Sum along columns (less cache-friendly)

// Best: Transpose for column operations
auto at = transpose(a);
auto col_sums = sum(at, 1);  // Now cache-friendly
```

### Memory Alignment

```cpp
// Arrays are automatically 64-byte aligned for SIMD
auto a = random::normal({1000, 1000});
assert(reinterpret_cast<uintptr_t>(a.data<float>()) % 64 == 0);
```

### Large Page Support

```cpp
// Enable large pages for huge allocations (Linux)
cpu::enable_huge_pages(true);

// Create large array with huge pages
auto huge = random::normal({100000, 100000});
```

## Parallelization Patterns

### Automatic Parallelism

Most operations are automatically parallelized:

```cpp
// Element-wise ops: parallelized across elements
auto c = add(a, b);  // Parallel

// Reductions: parallel with tree reduction
auto s = sum(a);  // Parallel

// Matmul: uses parallel BLAS
auto m = matmul(a, b);  // Parallel
```

### Nested Parallelism Control

```cpp
// Disable nested parallelism for outer-loop parallelization
cpu::set_nested_parallelism(false);

#pragma omp parallel for
for (int i = 0; i < batch_size; i++) {
    // Each iteration runs single-threaded
    auto result = process(batch[i]);
}
```

## Performance Tips

### 1. Batch Small Operations

```cpp
// Slow: overhead dominates
for (int i = 0; i < 1000; i++) {
    auto result = add(small_arrays[i], 1.0f);
}

// Fast: batched
auto batched = stack(small_arrays, 0);
auto result = add(batched, 1.0f);
```

### 2. Avoid Unnecessary Copies

```cpp
// Creates a copy
auto b = a + 0.0f;

// In-place (no copy)
add_(a, 1.0f);  // If supported
```

### 3. Use Contiguous Arrays

```cpp
// After slicing, array may be non-contiguous
auto slice = a[{Slice(0, 100), Slice(0, 100)}];

// Make contiguous for faster operations
auto contig = contiguous(slice);
auto result = matmul(contig, contig);  // Faster
```

### 4. Profile Thread Scaling

```cpp
#include <chrono>

// Find optimal thread count
for (int t = 1; t <= 16; t++) {
    cpu::set_num_threads(t);

    auto start = std::chrono::high_resolution_clock::now();
    auto result = matmul(a, b);
    eval(result);
    auto end = std::chrono::high_resolution_clock::now();

    auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(end - start);
    std::cout << t << " threads: " << ms.count() << " ms\n";
}
```

## Comparison with GPU

When to use CPU vs GPU:

| Scenario | Recommended |
|----------|-------------|
| Small arrays (&lt;10K elements) | CPU |
| Large batch operations | GPU |
| Sequential dependencies | CPU |
| Parallel independent ops | GPU |
| Memory-bound operations | CPU (unified memory) |
| Compute-bound operations | GPU |

### Hybrid Execution

```cpp
// Preprocessing on CPU
auto data = load_data();  // CPU
auto normalized = preprocess(data, Device::cpu);

// Heavy compute on GPU
eval(normalized);  // Ensure ready
auto features = model_forward(normalized, Device::gpu);

// Postprocessing on CPU
eval(features);
auto result = postprocess(features, Device::cpu);
```

## Troubleshooting

### Slow Performance

```cpp
// Check thread count
if (cpu::get_num_threads() == 1) {
    cpu::set_num_threads(std::thread::hardware_concurrency());
}

// Verify BLAS is linked
#ifdef MLX_HAS_BLAS
    std::cout << "BLAS available\n";
#else
    std::cout << "Warning: No BLAS, using slow fallback\n";
#endif
```

### High Memory Usage

```cpp
// Clear evaluation caches
clear_cache();

// Process in chunks
for (auto chunk : split(huge_array, chunk_size)) {
    process(chunk);
    eval();  // Free intermediates
}
```
